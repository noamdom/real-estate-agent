# Analyst Node Prompt — v1 (Baseline)

**File:** `nodes.py → analyst_node`
**Model:** `gpt-4.1-nano`, `temperature=0`
**Status:** current production prompt

---

## System Prompt

```
You are a senior property analyst for a real estate embassy.
Your job is to evaluate property listings and provide acquisition recommendations.
Reply with a JSON object containing exactly these keys:
  market_context, property_assessment, pricing_opinion, recommendation, expected_timeline
Keep each value to 1–3 sentences. Be factual. Do not invent prices or legal claims.
recommendation must start with one of: BUY | NEGOTIATE | RENT | PASS.
```

## User Prompt Template

```
Property: {property_type} in {location}
Size: {size_sqm} sqm, {num_rooms} rooms
Asking price: {price_asking} NIS
Condition: {condition}
Intent: {intent}

Comparable listings:
{comps_text}
```

Where `comps_text` is built as:
```
- {location}, {size_sqm}sqm, sold {price_sold:,.0f} NIS, {days_on_market} days on market (similarity {similarity_score})
```

---

## What's Missing / Failure Modes

### Context not passed to LLM
- `description` — the free-text listing description from the original payload is **never included**. The LLM has no textual context about the property beyond the structured fields.
- `agent_name` — not passed (minor, but useful for audit trail in multi-agent scenarios).
- `similarity_score` quality signal — the model gets scores like `0.73` with no frame of reference. It doesn't know whether that is a strong or weak match.

### Pricing analysis is implicit
- No instruction to calculate **price per sqm** (`price_asking / size_sqm`) and compare it to comps.
- Model may produce a pricing_opinion that sounds confident but is not grounded in actual arithmetic.
- Comps include `price_sold` but the prompt doesn't explicitly say "use this to assess whether the asking price is fair".

### `days_on_market` is unused in practice
- The field is in the comp data but the prompt never tells the model what to do with it.
- `expected_timeline` output is rarely driven by this signal — model tends to generic statements.

### Recommendation lacks justification
- `BUY | NEGOTIATE | RENT | PASS` is binary. No instruction to append a deviation percentage or reason.
- Example of what's missing: `NEGOTIATE — asking price is ~14% above comp average per sqm`.

### 1–3 sentence cap can be too tight
- `pricing_opinion` often needs more room to reference multiple comps — the cap forces compression that loses nuance.
- `market_context` is frequently boilerplate with no locality specifics because the model has no market data beyond the comps.

### No fallback instruction
- When `comps` is empty (`"No comparable listings found."`), the model still tries to produce a pricing_opinion, often fabricating market norms.
- The prompt doesn't say "if no comps are available, state that explicitly and lower confidence in the pricing opinion".

### Model is gpt-4.1-nano
- Very small model. Complex multi-step reasoning (compare 3 comps, compute price/sqm ratios, weigh days_on_market) may exceed its reliable capability at temperature=0.
- Consider whether analyst_node warrants a larger model.

## What to fix in v2

1. Add `description` to user prompt — gives the model textual context.
2. Add explicit pricing instruction: "Calculate price per sqm for subject and each comp. State the % deviation."
3. Add fallback instruction for empty comps.
4. Explain what `days_on_market` means and instruct model to cite it in `expected_timeline`.
5. Add similarity score guidance: "Comps with similarity < 0.6 are weak matches — note this in your assessment."
6. Expand `recommendation` format: `BUY | NEGOTIATE | RENT | PASS — <one-line reason with % if pricing>`.
