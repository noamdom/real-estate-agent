# RAG Node Prompt — v1 (Baseline)

**File:** `nodes.py → rag_node`
**Status:** No LLM call — pure vector search only

---

## Current Behaviour

The node builds a plain query string, embeds it, and returns raw Pinecone matches.
There is no LLM synthesis step.

### Query string (hardcoded template)
```python
query_text = (
    f"{norm.get('property_type', '')} in {norm.get('location', '')} "
    f"{norm.get('size_sqm', '')} sqm {norm.get('num_rooms', '')} rooms"
)
```

Example: `"apartment in Tel Aviv 85 sqm 4 rooms"`

### Return value (per comp)
```json
{
  "id": "...",
  "location": "...",
  "price_sold": 2100000.0,
  "size_sqm": 82.0,
  "days_on_market": 21,
  "similarity_score": 0.8312
}
```

---

## What the guideline expects

> Build a FastAPI server that receives a property description, embeds it [...], generates
> a short insight using Llama.cpp with a GGUF model file.

The current implementation satisfies retrieval but **skips the synthesis step** entirely.
The comps are passed as raw data into `analyst_node`, which is then responsible for both
interpreting the comps AND producing the full analysis — that's too much for one prompt.

---

## Failure Modes

### No insight generated
- `rag_node` returns raw numbers. `analyst_node` must infer all meaning from them.
- The RAG-specific prompt engineering task (guideline §4.1) is untouched.

### Query doesn't encode condition or price range
- Two properties with identical type/location/size but different condition (new vs. poor) get the same embedding — potentially retrieving irrelevant comps.
- Price range is excluded — a luxury villa may match a budget apartment on type+location alone.

### No quality signal to downstream
- When all similarity scores are low (e.g., < 0.5), the downstream analyst still receives 3 comps and treats them as authoritative.
- No mechanism to signal "these comps are poor matches — low confidence in pricing opinion".

### Silent failure hides bad retrieval
- On Pinecone error, `comps = []` and the analyst node receives `"No comparable listings found."` — this is logged but not surfaced to the caller in any structured way.

---

## What to add in v2

1. Add an LLM synthesis step after retrieval — produce a short `rag_insight` string that:
   - Cites which comp is closest and why.
   - Calculates average price/sqm across comps.
   - Notes if average days_on_market suggests a fast or slow sub-market.
2. Enrich the query string to include `condition` and a price bucket.
3. Pass `rag_insight` through state and surface it in the final output.
4. Add a `retrieval_confidence` field: "strong" (avg score ≥ 0.75) / "moderate" / "weak" (< 0.5).

### Candidate system prompt for v2 synthesis step
```
You are a property market analyst assistant.
You have retrieved {n} comparable listings from a vector database.
Summarise the comparable data in 2–3 sentences:
  - State the average price per sqm across the comps.
  - State the typical days on market in this sub-market.
  - Note whether the comps are strong (similarity ≥ 0.75) or weak matches.
Do not invent data. If no comps were found, say so directly.
```
